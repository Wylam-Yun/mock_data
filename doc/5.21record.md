# Mini-Agent SOP 合规检测实验记录（5.21）

## 实验目标

延续 5.20 实验，在消除数据集中的先验知识泄露后，重新测试 Mini-Agent 的 SOP 合规检测能力。

---

## 一、数据集修复：消除先验知识泄露

5.20 实验发现数据集中存在多个先验知识泄露问题，Agent 可以通过非语义方式直接定位有效日志。本次对所有事件 JSON 文件进行了系统性排查和修复。

### 发现的泄露项

| 泄露项 | 位置 | 影响 |
|--------|------|------|
| `related_sop_ref` 字段 | event_clean.json / event_with_noise.json | Agent 直接靠该字段精确匹配 SOP 步骤，不需要语义理解 |
| `log_id` 编号规则 | event_with_noise.json / event_noise_paraphrased.json | 有效日志 EVT-000xx，噪声日志 EVT-1xxxx，Agent 看 ID 即可区分 |
| `seq_id` 编号规则 | 同上 | 有效日志 seq_id <= 100，噪声日志 seq_id > 100 |
| 元数据计数 | metadata 中的 useful_logs_count / noise_logs_count | 直接暴露有效/噪声数量 |
| `dataset_id` 版本暗示 | 三个文件不统一 | 暗示数据集版本差异 |

### 修复措施

1. **删除 `related_sop_ref`**：从所有事件 JSON 中移除，仅保留在 `event_paraphrase_mapping.json` 作为 ground truth
2. **统一 `log_id` 编号**：将 5200 条日志全部打乱，重新分配连续编号 EVT-00001 ~ EVT-05200，有效/噪声不再有 ID 规律可循
3. **统一 `seq_id`**：改为连续 1~5200，与 log_id 同步
4. **删除元数据计数**：移除 useful_logs_count / noise_logs_count
5. **统一 `dataset_id`**：三个文件统一为 `mock_obs_bill_extend_sop_event_logs`
6. **修复 ID 分配 bug**：之前的脚本通过 event_time+actor 匹配有效日志，但噪声日志可能共享相同的时间和操作人，导致 89 条被误标为有效。改用复合键（event_name + platform + params.TargetCluster + alarm_id + component + metric）精确匹配，确认恰好 52 条有效

### 修复验证

- log_id 范围：EVT-00001 ~ EVT-05200，连续无断档
- seq_id 范围：1 ~ 5200，连续无断档
- 有效/噪声日志字段结构完全一致，无任何字段可区分
- 平台、角色、操作人分布：有效与噪声日志一致
- ground truth 映射文件 52 条全部验证通过

## 四、关键发现

### 发现一：高度特异关键词 = 轻松命中

SOP-EVT-003 的关键词 `1043006-bill-progress-delay` 是全局唯一的字符串，两个实验都直接精确定位到 1 条正确日志。只要 SOP 步骤描述中包含高度特异的关键词（如告警 ID、特定命令、唯一标识符），Agent 就能通过 grep/Select-String 轻松找到。

### 发现二：非高度特异关键词 = 搜不到或噪声淹没

SOP-EVT-001 和 SOP-EVT-002 的关键词（"未处理告警"、"SLA 成功率"）在 5200 条日志中大量出现。Agent 的应对策略是**组合多个弱特征做交集过滤**（平台 + 集群 + 关键词），但这种方式：

- **搜不全**：实验五中 Select-String 跨行匹配失败，直接漏掉 SOP-EVT-002
- **噪声太多**：实验六中匹配到 155 条和 89 条结果，无法区分哪条是真正的有效日志

### 发现三：Agent 的匹配逻辑始终是关键词 + 字段过滤，没有语义理解

两个实验中 Agent 的匹配方式本质相同：

1. 从 SOP 提取关键词（告警、SLA、1043006 等）
2. 在 JSON 中搜索包含这些关键词的日志
3. 用平台、集群等字段做二次过滤

这个流程中没有任何一步涉及语义理解。Agent 不会：
- 理解 SOP 步骤的约束条件（如"未处理"告警 vs 所有告警）
- 理解参数约束（如必须是"目标集群"的 SLA，而非所有集群）
- 用排除法缩小范围
- 利用 SOP 的阶段/上下文信息

### 发现四：语义改写确实增加了难度，但不是通过"逼迫语义理解"实现的

实验六比实验五准确率更低（1/3 vs 2/3），但原因不是 Agent 被迫做语义理解失败了，而是：
- 措辞一致时，`event_name: "查询未处理告警"` 与 SOP 描述完全匹配，Agent 能精确命中
- 语义改写后，`event_name: "拉取集群活跃告警"` 与 SOP 描述不同，Agent 的关键词搜索匹配不到，只能退化到更宽泛的条件过滤，噪声更多

## 五、综合结论

| 实验 | 数据 | 先验泄露 | 有效步骤准确率 | 核心匹配方式 |
|------|------|---------|--------------|-------------|
| 一~三 | 52 / 5200 条 | related_sop_ref 存在 | 3/3 | related_sop_ref 精确匹配 |
| 四 | 5200 条改写 | related_sop_ref 改写 | 1/3（搜不全） | read_file 被动阅读 |
| 五 | 5200 条（修复后） | 已消除 | 2/3 | 关键词 + Select-String |
| 六 | 5200 条改写（修复后） | 已消除 | 1/3 | 关键词 + Python 脚本过滤 |

### 结论

**Agent 的 SOP 合规检测能力高度依赖关键词特异性**。当 SOP 步骤包含全局唯一标识符时，Agent 能精准定位；当关键词在噪声中大量出现时，Agent 无法通过语义理解区分有效日志和噪声日志，只能靠弱特征交集过滤，结果要么搜不全，要么噪声淹没有效信号。

消除先验泄露后，Agent 的匹配方式从"结构化字段精确匹配"退化为"关键词 + 字段值过滤"，准确率从 3/3 下降到 1/3~2/3。这说明之前的高准确率完全依赖先验知识，而非真正的语义理解能力。
