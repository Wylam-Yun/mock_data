# Mini-Agent SOP 合规检测实验记录

## 实验目标

验证 Mini-Agent 能否基于 SOP 文件和事件 JSON 文件，自动判断某个 SOP 动作是否存在执行证据，并截取对应事件。

## 实验任务（固定）

> 当前存在两份文件，分别是 SOP 文件 `D:\Weilin_Workspace\mock_data\sop.md` 和事件数据文件 `D:\Weilin_Workspace\mock_data\event_*.json`。其中 SOP 文件中包含"告警检测"操作，事件 JSON 中包含相关操作数据。要求 Agent 判断"告警检测"这个操作是否被执行；如果已执行，则截取对应事件证据。

SOP 中"告警检测"包含 5 个子步骤：SOP-EVT-001（查询未处理告警）、SOP-EVT-002（查看 SLA 成功率）、SOP-EVT-003（查询 1043006-bill-progress-delay 告警）、SOP-EVT-004（联系 oncall 确认）、SOP-EVT-005（告警检测验证）。

---

## 实验一：纯净数据 + 精确匹配

- **数据文件**：`event_clean.json`（52 条有效日志，无噪声）
- **有效日志措辞**：与 SOP 完全一致
- **related_sop_ref**：保留原始路径（如 `01操作检测/1告警等检测/告警检测/操作步骤`）

### 结果

Agent 正确找到 LOG-001、LOG-002、LOG-003 三条证据，分别对应 SOP-EVT-001/002/003。判断"告警检测部分执行（3/5 步骤）"，SOP-EVT-004 和 005 未找到执行记录。

### 匹配方式

Agent 主要依赖 `related_sop_ref` 字段做精确字符串匹配，筛选出该字段包含"告警检测"的事件，再用 `event_name` 做二次校验。

### 评价

结果正确，但匹配方式本质是字段检索 + 规则匹配，不涉及语义理解。

## 实验二：含噪声数据 + 精确匹配

- **数据文件**：`event_with_noise.json`（5200 条：52 有效 + 5148 噪声）
- **有效日志措辞**：与 SOP 完全一致
- **related_sop_ref**：保留原始路径；噪声日志该字段为 null
- **噪声特征**：按 SOP 阶段生成上下文相关噪声，仅 lisi/zhangsan 两人，覆盖 17 个平台

### 结果

Agent 正确找到 LOG-001/002/003，结论与实验一一致。噪声日志未造成干扰（因噪声的 `related_sop_ref` 为 null）。

### 过程问题

Agent 陷入严重循环：62 个 REQUEST/RESPONSE 周期、6360 行日志。反复重复相同的 thinking chain（读文件 → PowerShell 编码报错 → 写 Python 脚本 → 修复 NoneType → 编码问题 → 重试），context window 溢出后 summary 重建又从头开始。

### 评价

结果正确，但暴露了 Agent 的工程问题：PowerShell 编码兼容性差、无循环终止机制、context 管理失败。

## 实验三：语义改写 v1（event_name / detection_log 改写，related_sop_ref 保留）

- **数据文件**：`event_noise_paraphrased.json`（v1 版，5200 条）
- **有效日志措辞**：`event_name`、`detection_log`、`params` 做了语义改写（如"查询未处理告警" → "拉取集群活跃告警"，"检测日志" → "运行日志"）
- **related_sop_ref**：保留原始路径，未改写

### 结果

Agent 正确找到 LOG-001/002/003，结论一致。但 LOG-002 的 event_name 写错了（报告写"获取SLA成功率数据"，实际数据是"获取 SLA 可用性数据"），存在轻微 hallucination。

### 匹配方式

Agent 仍然靠 `related_sop_ref` 精确匹配找到有用日志。虽然 `event_name` 和 `detection_log` 被改写了，但 `related_sop_ref` 没变，所以匹配没有受到影响。

### 评价

语义改写没有生效——因为 `related_sop_ref` 保留了原始路径，Agent 没有被逼到需要语义理解的地步。该实验未能真正测试语义匹配能力。

---

## 实验四：语义改写 v2（全部字段改写，含 related_sop_ref）

- **数据文件**：`event_noise_paraphrased.json`（v2 版，5200 条）
- **有效日志措辞**：`event_name`、`detection_log` 继续改写
- **related_sop_ref**：全新命名体系（如 `01操作检测/1告警等检测/告警检测/操作步骤` → `一、变更前检查/1.1 告警排查/告警排查/执行记录`），共 25 条路径全部改写

### 结果

Agent 只找到 LOG-001，漏掉 LOG-002 和 LOG-003。结论为"告警检测操作已执行"，但只基于 1 条证据，且未说明 SOP-EVT-004/005 缺失。

### 过程

Agent 仅用 6 轮（177 行日志）就给出了结论。读取了 sop.md 后，尝试读取 3MB 的 JSON 文件，但 context window 放不下全部 5200 条日志，只看到了文件开头的几十条，发现 LOG-001 就停止了搜索。

thinking 中提到了 3 条 LOG，但实际 output 只写了 LOG-001，thinking 与 output 不一致。

### 评价

结果不完整，存在幻觉。`related_sop_ref` 改写确实起到了作用——Agent 不再能靠精确匹配轻松找到所有有用日志，但它也没有能力通过语义理解补上。暴露了两个核心问题：

1. **大文件处理能力不足**：3MB JSON 无法在 context window 内完整呈现，Agent 不知道自己没看完
2. **缺乏主动搜索工具调用**：没有使用 grep/bash 等工具在 JSON 中搜索关键词，仅依赖 read_file 的被动阅读

---

## 综合结论

| 实验 | 数据 | related_sop_ref | 找到 LOG | 结论正确性 | 核心匹配方式 |
|------|------|-----------------|----------|------------|-------------|
| 一 | 52 条纯净 | 原始 | 001/002/003 | 正确 | related_sop_ref 精确匹配 |
| 二 | 5200 条含噪声 | 原始 | 001/002/003 | 正确 | related_sop_ref 精确匹配 |
| 三 | 5200 条改写 v1 | 原始 | 001/002/003 | 基本正确（轻微 hallucination） | related_sop_ref 精确匹配 |
| 四 | 5200 条改写 v2 | 全部改写 | 只有 001 | 不完整 | 未完成搜索 |

### 关键发现

1. **Agent 严重依赖 `related_sop_ref` 结构化字段**：只要该字段保留原始命名，Agent 就能通过精确匹配找到证据，不需要任何语义理解能力
2. **`related_sop_ref` 改写后 Agent 能力骤降**：无法靠精确匹配，也没有退化到语义理解，而是直接搜不全或放弃搜索
3. **大文件是硬伤**：Agent 无法在单次 read_file 中处理 5200 条日志的 JSON，又不会主动使用 grep/bash 等工具做定向搜索
4. **工程稳定性差**：PowerShell 编码问题、无循环终止、context 溢出后无限重复，消耗大量 token
5. **thinking 与 output 不一致**：thinking 中分析了 3 条 LOG，output 只输出 1 条

### 后续改进方向

- Agent 需要具备对大 JSON 文件的定向搜索能力（grep/bash 工具调用），而非仅靠 read_file 被动阅读
- 需要引入语义检索 / embedding 匹配机制，当结构化字段不可用时能退化到语义理解
- 需要循环终止保护和 context 溢出后的主动压缩策略
